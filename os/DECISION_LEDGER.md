# DECISION LEDGER

Significant architectural, strategic, or venture decisions are recorded here with their rationale so future changes can be evaluated against the original assumptions.

## Log
* **Undated (pre-2026-06-29) — Implemented Stateless Architecture:** Decided to make the repository the single source of truth and use the AI as a stateless processor. Rationale: Eliminates the risk of losing context if chat history is cleared or AI models are switched.
* **2026-06-29 — Core-OS-first sequencing:** Build the Core OS (PRODUCT, ROADMAP, CHANGELOG, ASSUMPTIONS, LESSONS_LEARNED) before expanding Venture 001. Rationale: treat Venture OS as the product and Venture 001 as its proving ground; establish the permanent operating system before venture-specific operational assets.
* **2026-06-29 — Adopted Venture States lifecycle:** Replaced the coarse "Active" status with a 9-state lifecycle (Idea → Research → Validation → Prototype → Execution → Revenue → Scale → Maintenance → Archived). Rationale: state dictates appropriate work and the evidence bar; advancement requires evidence, ideally a cleared payment.
* **2026-06-29 — Established Core Primitives + Single Responsibility:** Promoted Founder, Reality, AI Reasoning Engine, Ventures, Knowledge, Assets, and Core Engines to first-class primitives, each with one home and one single responsibility. Rationale: enables compounding and cross-venture scale while preventing architectural drift. Core Engines admitted only under the Extraction Rule (a process must be proven at least twice before it is abstracted into an engine).
* **2026-06-29 — Froze PRODUCT.md v1.0:** Locked the product spec after a formal design review. Rationale: stabilize the foundational definition; modify only if future execution demonstrates a structural weakness.
* **2026-06-29 — Hosted Venture OS on GitHub (public):** Created `github.com/bhanu-thakur/venture-os` (Public, per founder's explicit choice) and seeded the full Core OS; GitHub is now the authoritative source of truth, with commit + read-back verification as standing protocol. Rationale: enable real version control and the repository-driven architecture. Risk noted: public visibility — no secrets/sensitive data in repo.
* **2026-06-29 — Technical architecture = static repository-driven PWA on GitHub Pages:** Vanilla HTML/CSS/JS, zero backend, zero infra cost, offline-capable, mobile-first; the repository is the database; the app is a thin render layer. Rationale: minimal cost, AI-friendly, no lock-in, aligns with the Indian-market zero-cost reality. Trade-offs: no dynamic queries, hash routing on Pages, client-side Markdown rendering, hand-maintained `index.json` (drift mitigated by a generator + verification). Framework rejected (fails the justification test); only earned dependency is a vendored Markdown parser.
* **2026-06-29 — Architecture philosophy: explicit ownership + reconstructability (TECHNICAL_ARCHITECTURE v2.0 frozen):** Established a single-owner model for every concern (Repository = durable memory; AI Reasoning Engine owns reasoning, never state; Founder owns decisions/execution; Reality owns validation) and a hard reconstructability guarantee — the entire system can be rebuilt from the repository alone, surviving loss of chat, AI vendor, host, or app. Design system declared a replaceable Application-Layer dependency. Rationale: durability and no catastrophic lock-in across years and tool changes. See ADR-0010, ADR-0011.
* **2026-06-29 — Architecture review → TECHNICAL_ARCHITECTURE v2.0 + ADRs:** Restructured the technical architecture around five layers (Knowledge/Domain/Application/Reasoning/Execution), corrected the repository-vs-database framing (repository = source of truth; app derives runtime state), added a Domain Model and an explicit Compounding Loop, and adopted Semantic Versioning. Introduced `ARCHITECTURE_DECISIONS.md` for immutable, numbered **technical** ADRs — this ledger now holds **strategic/venture/business** decisions only; technical rationale moves to the ADR file. Rationale: separation of concerns and decade-scale maintainability; the two ledgers have different lifecycles and audiences. See ADR-0004, ADR-0006, ADR-0008, ADR-0009.
* **2026-06-29 — Card Atlas adopted as canonical design system:** `card-atlas-design-system.html` is the single source of visual truth — copied verbatim, tokens-only, default forest-green theme. Consumed, never redesigned. Compatibility adaptations for offline (self-hosted fonts; keep `color-mix()`) are explicitly not visual changes. Rationale: consistency, zero design debt, GitHub-Pages-ready.
* **2026-06-29 — Adopted holding-company repository structure (frozen before Venture 001):** Top-level `os/ · founder/ · ventures/ · library/ · static/`; ventures first-class and additive; State (on venture README) separated from Mission (daily); business vs execution split; reusable assets promoted to `library/`; venture data Markdown-first. Rationale: scale to 10–20 ventures with no future restructuring, done now while the repo has zero ventures (cheapest moment). See ADR-0015, ADR-0016, ADR-0017.
