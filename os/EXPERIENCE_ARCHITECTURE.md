# EXPERIENCE ARCHITECTURE

**Version:** 1.0
**Status:** LOCKED — Frozen at v1.0 (2026-06-29)
**Governs:** How the Venture OS runtime turns repository truth into clarity, confidence, focus, and momentum. Implementation-independent. `PRODUCT.md` defines what the system is; `TECHNICAL_ARCHITECTURE.md` defines how it runs; this defines **how it feels to operate**.

> The repository stores truth. The runtime creates experience. The founder operates a company — never browses Markdown.

---

## 1. THE PRODUCT IS THE DAILY LOOP (not a set of pages)

```
Mission → Real-world execution → Capture → Repository updated →
Progress becomes visible → Review → Better decisions → Tomorrow's Mission
```

Each turn must **compound** (a lesson banked, an asset created, a state advanced), never merely repeat. Everything in the runtime serves this loop or it is not built.

## 2. TWO TIERS — AND ONLY TWO

* **The Cockpit (one daily home): Mission.** The only thing a founder navigates to each morning.
* **Reference (summoned, never a morning menu): Knowledge · Review · Portfolio · Founder · Assets** — reached through the **Command Palette**, built only when their repository content exists.

## 3. MISSION — THE COCKPIT

Everything that pulls daily lives here, in descending order of volume:

1. **Orientation** — venture · State (quiet).
2. **A decision, if one waits** — surfaced here so better decisions interrupt the morning, not a tab.
3. **The win** — today's objective (the headline).
4. **The one action** — the single highest-leverage move (the focal element; "Begin").
5. **Since last time** — what actually moved (state, proof, revenue) — the honest daily-return pull.
6. **Capture** — a one-line, paste-ready prompt to bank yesterday's outcome into the repository (the runtime generates; the founder commits — it never persists business data in-app).
7. **On demand (below the fold):** the journey path, revenue line, why it matters, proof.

## 4. REFERENCE (summoned via Command Palette; depth, not daily)

* **Knowledge** — the quiet shelf; retrieval, not browsing.
* **Review** — playback of history + the decision log, ending on takeaways.
* **Portfolio** — the venture constellation by State; choose focus. *Activate at ≥2 ventures.*
* **Founder** — profile, skills, gear. *Activate when `founder/profile.md` exists.*
* **Assets** — reusable library; promotion from ventures. *Activate when assets exist.*

**Cross-cutting: Command Palette** — jump to any venture, document, reference view, or action in one keystroke. The backbone of calm at scale and the only navigation needed beyond Mission.

**Removed/merged:** *Decisions* (standalone) → split into Mission's decision beat + Review's log. *Observe, Execute, Learn* → removed/folded (execution happens in the world; learning lives in Knowledge).

## 5. EMOTIONAL GOALS (the ship test)

Clarity · Confidence · Focus · Momentum · Progress · Calm-at-scale. A surface that advances none of these on arrival does not belong on arrival.

## 6. INFORMATION HIERARCHY (every surface)

Single statement → single action → quiet context → depth on demand. Arrival shows the first two; the rest rewards a tap or a keystroke.

## 7. INTERACTION PHILOSOPHY

Read-mostly: the runtime points; the founder acts in the world; the founder records in the repository. Jump-anywhere via the palette. No modals, no business-data forms; Capture generates-and-commits, never in-app persistence. Reversible (the repository is the record). Instant, quiet, offline-first.

## 8. EXPERIENCE LIFECYCLE

The daily loop of §1. Health is measured by whether each cycle **compounds** — proof accumulated, a lesson banked, a state advanced — not by activity.

## 9. DESIGN PRINCIPLES (frozen filter)

Founder-state over pages · the loop is the product · one daily home (Mission), everything else summoned · one primary action per surface · close the loop (orient → act → capture → progress) · progress earned and honest · the runtime presents, the repository records · content before cockpit (views cheap, systems earned) · every surface reduces a Four Cost.

## HOW THIS DOCUMENT EVOLVES

Frozen at v1.0. From here, the experience evolves through **implementation and real usage**, not further foundational redesign. Changes are made only when real daily use proves a principle wrong, and are logged in `CHANGELOG.md` with rationale in `DECISION_LEDGER.md`.
